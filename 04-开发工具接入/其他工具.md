# 其他工具

更新时间：2026-04-16

本页只保留可判断、可复现的接入原则。

如果某个工具没有单独列在文档目录里，请先判断它真实支持的是哪一种协议，再选择对应文档，不再默认认为“所有带 OpenAI 字样的工具都一定可接”。

## 一、先判断工具属于哪一类

### 第一类：支持自定义 OpenAI Compatible 网关

这类工具通常具备以下特征：

- 可以填写自定义 `Base URL`
- 可以填写自定义 `API Key`
- 可以填写自定义 `Model`
- 实际调用的是 `chat/completions`、`responses`、`embeddings` 等 OpenAI 风格接口

这类工具请直接参考：

- [OpenAI格式（支持各大原厂模型）](../01-AI大模型API/OpenAI格式（支持各大原厂模型）.md)

### 第二类：工具只支持非 OpenAI 专属协议

这类工具通常具备以下特征：

- 不能直接填写 OpenAI Compatible 网关
- 要求固定的专属协议或专属请求格式
- 对接前需要先确认当前文档站是否公开承诺该协议

如果当前公开文档没有单独列出该协议页，优先选择：

- 使用支持 OpenAI Compatible 的接入方式
- 或直接改用原生 HTTP Request
- 或参考已单独提供的工具页，例如 [Claude Code](Claude%20Code.md)

### 第三类：支持原生 HTTP Request

这类工具不要求内置某个平台节点，只要能发 HTTP 请求即可。

这类工具可以直接调用 ApiHalo 的公开接口，是最稳的通用方式。

### 第四类：不支持自定义网关

如果工具只允许固定官方供应商、固定登录方式，或不允许修改请求地址，那么它就不属于可保证的直连接入工具。

这种情况下，请不要强行把 ApiHalo 地址填进去。

## 二、通用排查顺序

1. 先确认工具是否明确支持自定义请求地址
2. 再确认它走的是哪一种协议
3. 再确认模型是否来自 `/v1/models`
4. 再用最小请求做一次真实测试

## 三、最稳的做法

如果你无法确定某个工具到底走什么协议，优先按以下顺序处理：

1. 先用 `curl` 验证 ApiHalo 接口本身可通
2. 再确认该工具是否支持自定义 OpenAI 网关或原生 HTTP
3. 只有在工具明确支持时，再写入工具配置

## 四、统一接入原则

无论具体工具是什么，只要它是通过 ApiHalo 的公开接口接入，客户侧都应始终使用：

- ApiHalo 的域名
- ApiHalo 的 API Key
- `/v1/models` 返回的真实模型 ID

## 五、后续更换上游是否受影响

不受影响。

因为客户侧文档承诺的是 ApiHalo 的公开接口口径，而不是某个特定上游平台的私有配置方式。
